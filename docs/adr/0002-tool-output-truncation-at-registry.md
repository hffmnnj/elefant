# ADR-0002: Tool output truncation at registry boundary

**Status:** Accepted
**Date:** 2026-04-28
**Workflow:** agent-harness-research
**Recommendation:** [R5](../../markdown-db/07-research/HARNESS-RECOMMENDATIONS.md)
**MH:** MH8

## Context

Tool output truncation was previously implemented inconsistently per-tool. For example, `webfetch` wrote full content to `/tmp` and truncated at ~50k chars, while other tools returned output as-is. The ToolRegistry always converted tool output to a single string and emitted it without a centralized size policy (`src/tools/registry.ts:50-60`, `:79-92`). This inconsistency created context budget surprises and made it difficult to reason about maximum output sizes. Reference systems (OpenCode, Claude Code) centralize truncation as a registry/service policy so all tools inherit the same "context safety" behavior.

The gap lived in the absence of a registry-level truncation boundary. Without it, large tool outputs could consume disproportionate context, and tools had no consistent guidance on output size expectations.

## Decision

Added a `truncateToolOutput()` helper function in `src/tools/registry.ts` that uniformly truncates tool output to a configurable maximum (default 100,000 chars). The truncation:

1. Accepts `content`, `maxChars` (default 100,000), and optional `maxLines` parameters
2. Truncates by line count first if `maxLines` is specified
3. Then truncates by character count if content exceeds `maxChars`
4. Returns `{ content: string; wasTruncated: boolean }` to signal truncation
5. Appends a consistent truncation notice: `"\n\n[Output truncated: showing first N chars. Use grep/glob/read to retrieve specific parts.]"`

Truncation is applied uniformly to:
- Successful tool outputs (before `tool:after` emit and return)
- Error tool outputs generated by the registry (validation errors, exceptions)

Tool-specific "full output stored at ..." notices are preserved before registry truncation, so users can still locate the full output if needed.

## Consequences

### Positive
- All tools inherit the same output size policy without per-tool changes
- Context budget is protected by a consistent registry-level guard
- Truncation is transparent to the LLM (notice explains how to retrieve full output)
- Tool-specific "full output at ..." messages are preserved for user reference

### Negative / accepted trade-offs
- Large tool outputs are silently truncated; the LLM must use grep/glob/read to retrieve specific parts (this is intentional and documented in the notice)
- Truncation threshold (100,000 chars) is fixed; future work could make it configurable per tool or context window

### Backward compatibility
Existing tools continue to work unchanged. Tools that previously returned very large outputs will now be truncated, which is the intended behavior to protect context budgets. The truncation notice is new and will be visible to the LLM, which is the intended behavior.
